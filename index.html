Warhammer 40K 8th Edition List Builder

Status: On Hold - Progresses only in occassional, short, and rapid bursts of motivation

Currently this project is not getting any love as I haven't been playing 40K recently. Instead I've been trying out the new Kill Teams which is rather enjoyable. And in this case the BattleScribe app does quite a nice job of creating lists, so I have not felt the need to create my own.

I started this project for two reasons:

  • Paging back and forth through multiple rule books composing your army and hoping that you got all of the costs and special rules/dependencies correct is a real pain.
  • The list building applications that already exist are unstaisfactory (to me at least). The official app - Command Roster - is disgustingly under-featured and the best free contender - BattleScribe - is not the easiest/nicest to use.
So I decided to make my own web-based list builder. Originally this was planned to cater for all armies, however once I got round to translating the rules for the factions I play (Craftworlds, Harlequins, and Ynnari) I realised just how big of a task that would be. So, for now at least, this project will only support my factions, but should be able to support the others if I ever get around to adding their data.

2D Game Engine

Status: Started - Progressing Slowly

This project is (as the title suggests) a 2D game engine. This seems to be a popular choice of project for small teams and startups these days. This particular engine is intended to be used to create web-based games and is being developed in TypeScript.

Why would someone want a web-based game engine knowing the restrictions of the platform?
Honestly I've debated this with myself numerous times, however I keep settling back on web-based due to the appeal of accessibility and, to some degree, the challenge of seeing if I can make it work.

If you just want to make games then why not just use one of the existing engines?
I am rather biased against using libraries and engines. Mostly because of the excess code that comes them that you never use, the extra processing that you might not be making use of but runs anyway. But also because I like to understand the code I'm using, so unless the code-base is small enough for me to read through and understand then I prefer to write it myself. That way I learn more as well.
